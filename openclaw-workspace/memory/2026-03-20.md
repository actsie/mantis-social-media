## Patterns in What People Are Asking For / Complaining About

### Pattern 1: The iframe wall
Real agents hitting payment iframes they literally cannot penetrate. This isn't a protocol problem — it's a fraud-detection problem. Human checkout security is agent-hostile by design. @joozio documented it with a real experiment.

### Pattern 2: "I want to give my agent a card but don't know how"
Spontaneous demand from builders who see the use case clearly but have no clear path. AIVM_Network's "how do I give my agent one of these?" is the purest version of this. Not asking for a protocol — asking for a product.

### Pattern 3: Protocol confusion / fragmentation overwhelm
Builders are aware of x402, MPP, AP2, ACP. They don't know which to bet on. @GHOST PROTOCOL's "everyone and their grandma is dropping an agent payment SDK" captures the noise fatigue. Opportunity: Agent Card as the layer above the protocol that insulates you from the choice.

### Pattern 4: Controls are the missing piece
Multiple voices (@NORNR, @contofinance, @exploitless, @KirillTalai) are circling the same insight: raw payment rails exist but the governance/controls/audit layer doesn't. "Give agents a mandate before you give them money" — this is Agent Card's positioning in one sentence.

### Pattern 5: Speed and velocity
Card rails can't execute agent velocity (@matterhorn). Agents transacting autonomously for 15 hours (@KirillTalai). The human-designed payment system has rate limits, friction points, and latency that agents will break. Infrastructure needs to be designed for machine speed.
